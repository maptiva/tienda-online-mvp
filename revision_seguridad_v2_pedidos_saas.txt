## Revisión de Propuesta de Seguridad V2 — Pedidos SaaS

Revisado. La V2 cierra los tres vectores críticos identificados en la auditoría anterior. Aquí la mirada técnica:

---

### ✅ Lo que está correctamente resuelto

**Recálculo de totales en servidor** — implementado exactamente como se recomendó. El loop sobre `jsonb_to_recordset` cruzando contra la tabla `products` es el enfoque correcto. El total del cliente ahora es ignorado completamente para el cálculo real.

**`SET search_path = public, pg_temp`** — aplicado en el lugar correcto, como cláusula de la función y no dentro del bloque `BEGIN`. Esto es importante porque si estuviera dentro del cuerpo, habría una ventana mínima de vulnerabilidad antes de ejecutarse.

**Validación de propiedad cruzada (Cross-store)** — el `WHERE id = v_item.product_id AND user_id = (SELECT user_id FROM stores WHERE id = v_store_id)` es la defensa correcta contra el escenario de mezclar productos de tiendas distintas.

**Opacidad de respuesta** — devolver solo el UUID y nunca el ID numérico secuencial es una buena práctica. Los IDs secuenciales permiten enumerar registros.

---

### ⚠️ Observaciones menores que vale la pena revisar

**1. El descuento no está validado contra ningún límite**

    v_calculated_total := v_calculated_total - COALESCE(p_discount_applied, 0);

Un atacante puede enviar `p_discount_applied = 99999` y el total resultante sería negativo. Recomiendo agregar:

    IF p_discount_applied < 0 OR p_discount_applied > v_calculated_total THEN
      RETURN json_build_object('success', false, 'error', 'Descuento inválido');
    END IF;

**2. La subquery dentro del loop es un problema de performance**

    SELECT user_id FROM stores WHERE id = v_store_id

Esto se ejecuta una vez por cada ítem del pedido. Para un pedido de 10 productos son 10 queries idénticas. Mejor resolverlo una sola vez antes del loop y guardarlo en una variable `v_store_owner_id`.

**3. `p_client_total` se recibe pero nunca se usa**

El parámetro existe en la firma de la función pero no aparece en ninguna validación ni lógica. Podría eliminarse de la firma para reducir la superficie, o bien usarse para una verificación de consistencia con el total calculado (y rechazar si la diferencia es mayor a cierto umbral, como señal de alerta).

**4. El `EXCEPTION WHEN OTHERS` silencia todo**

Es correcto no exponer trazas al exterior, pero internamente deberías loguear el error real. Algo como `RAISE WARNING 'Error en create_public_order: %', SQLERRM;` antes del RETURN permite que el error quede en los logs del servidor sin exponerse al cliente.
